# TradingAgents 需求分析文档

## 1. 项目概述

### 1.1 项目背景

TradingAgents 是一个基于大语言模型（LLM）的多智能体金融交易分析框架。它模拟真实交易公司中的团队协作模式，将复杂的交易分析任务分解为多个专业化角色，通过 LLM 驱动的智能体之间的协作与辩论，评估市场状况并形成交易决策。

该项目由 Tauric Research 开源，旨在为金融 AI 研究提供一个模块化、可扩展的实验平台。

### 1.2 项目定位

- **研究导向**：不提供投资建议，交易表现因 LLM 模型、温度参数、交易时段、数据质量等因素而异
- **框架性质**：既可作为 Python 库嵌入代码使用，也提供交互式 CLI 工具
- **多提供商支持**：不绑定单一 LLM 提供商，支持 10+ 种 LLM 后端
- **多数据源**：支持 yfinance 和 Alpha Vantage 两种数据源，支持自动回退

## 2. 功能需求

### 2.1 核心分析流水线

系统必须实现一条完整的分析流水线，包含以下阶段：

#### 2.1.1 分析师阶段（Analyst Phase）

| 分析师 | 职责 | 数据来源 |
|--------|------|----------|
| **市场分析师（Market Analyst）** | 技术面分析，使用 13 种技术指标（SMA/EMA/MACD/RSI/布林带/ATR 等）识别交易模式和价格趋势 | OHLCV 数据、技术指标 |
| **社交媒体分析师（Social Media Analyst）** | 分析社交媒体和公众情绪，使用情绪评分算法衡量短期市场情绪 | 新闻数据 |
| **新闻分析师（News Analyst）** | 监控全球新闻和宏观经济指标，解读事件对市场状况的影响 | 公司新闻、全球新闻 |
| **基本面分析师（Fundamentals Analyst）** | 评估公司财务和业绩指标，识别内在价值和潜在风险 | 公司概况、资产负债表、现金流量表、利润表 |

#### 2.1.2 研究员辩论阶段（Researcher Debate Phase）

- **多头研究员（Bull Researcher）**：构建看涨论点，反驳空头观点
- **空头研究员（Bear Researcher）**：构建看跌论点，反驳多头观点
- 双方进行多轮结构化辩论（轮数可配置）
- **研究经理（Research Manager）**：综合辩论双方观点，产生结构化的投资计划（ResearchPlan），包含推荐评级、理由和战略行动

#### 2.1.3 交易决策阶段（Trading Phase）

- **交易员（Trader）**：将研究计划转化为具体的交易提案（TraderProposal），包含操作方向、推理、入场价、止损价、仓位规模

#### 2.1.4 风险辩论阶段（Risk Debate Phase）

- **激进分析师（Aggressive Analyst）**：支持高回报机会
- **保守分析师（Conservative Analyst）**：优先稳定性与风险缓解
- **中立分析师（Neutral Analyst）**：提出均衡观点
- 三方进行多轮结构化辩论（轮数可配置）

#### 2.1.5 决策阶段（Decision Phase）

- **投资组合经理（Portfolio Manager）**：综合风险辩论，输出最终的结构化投资决策（PortfolioDecision），包含评级、执行摘要、投资论据、目标价、时间展望
- 决策输出标准化为 **5 级评级体系**：Buy / Overweight / Hold / Underweight / Sell
- **评级提取**：PM 的结构化输出保证了 `**Rating**: X` 格式，评级提取为确定性正则启发式（不再需要 LLM 调用）

### 2.2 LLM 提供商支持

系统必须支持以下 LLM 提供商，并提供统一的抽象接口：

| 提供商 | 协议/方式 | 配置项 |
|--------|-----------|--------|
| OpenAI | Responses API (`/v1/responses`) | reasoning_effort |
| Anthropic | 原生 API | effort (extended thinking) |
| Google Gemini | 原生 API | thinking_level / thinking_budget |
| xAI (Grok) | OpenAI 兼容 API | — |
| DeepSeek | OpenAI 兼容 API + 思维模式特殊处理 | — |
| Qwen (DashScope) | OpenAI 兼容 API | — |
| GLM (智谱) | OpenAI 兼容 API | — |
| OpenRouter | OpenAI 兼容 API + 动态模型发现 | — |
| Ollama | OpenAI 兼容 API（本地） | — |
| Azure OpenAI | 企业级 API | 部署名、端点 |

#### 2.2.1 双轨道 LLM 策略

- **深度思考 LLM（Deep Thinking）**：用于研究经理和投资组合经理——需要综合多方辩论的复杂推理任务
- **快速思考 LLM（Quick Thinking）**：用于分析师、研究员、交易员、反思——工具调用和常规分析任务

#### 2.2.2 关键模型说明

- **OpenAI**：使用 Responses API（`/v1/responses`），支持 `reasoning_effort` 参数。所有模型家族（GPT-4.1、GPT-5.x）统一用此路径
- **DeepSeek**：thinking-mode 需回传 `reasoning_content`，否则 API 返回 400 错误。`deepseek-reasoner` 不支持 `tool_choice`，结构化输出自动降级为自由文本
- **Anthropic**：content 为数组格式，结构化输出走 tool-use 路径。空消息列表不允许，需占位 "Continue" 消息
- **Google**：Gemini 3 系列使用 `thinking_level`，Gemini 2.5 系列使用 `thinking_budget`

### 2.3 数据源支持

#### 2.3.1 数据分类

| 类别 | 包含工具 | 说明 |
|------|----------|------|
| core_stock_apis | get_stock_data | OHLCV 价格数据 |
| technical_indicators | get_indicators | 13 种技术指标 |
| fundamental_data | get_fundamentals, get_balance_sheet, get_cashflow, get_income_statement | 公司财务数据 |
| news_data | get_news, get_global_news, get_insider_transactions | 新闻与内幕交易 |

#### 2.3.2 数据源

- **yfinance**（默认）：免费，无需 API Key
- **Alpha Vantage**：需要 API Key，提供备用数据通路

#### 2.3.3 数据路由

- 支持按类别配置数据源（category-level）
- 支持按工具配置数据源（tool-level，优先级更高）
- 支持回退链：主数据源失败时自动尝试备用数据源
- 仅 `AlphaVantageRateLimitError` 触发回退，其他错误直接暴露

### 2.4 持久化与记忆

#### 2.4.1 决策日志（Memory Log）

- 位置：`~/.tradingagents/memory/trading_memory.md`
- 追加式 Markdown 文件，使用 `<!-- ENTRY_END -->` 作为条目分隔符
- 每个条目包含：日期、股票代码、评级、状态、决策内容、反思
- 两阶段模式：
  - **阶段 A**：每次运行结束追加一条 "pending" 状态的决策
  - **阶段 B**：下次同股票运行时，拉取实际收益数据，生成 LLM 反思，更新为已解析状态

#### 2.4.2 上下文注入

在投资组合经理的提示词中注入历史上下文：
- 最近 5 条同股票历史决策（完整内容）
- 最近 3 条跨股票历史教训（仅反思部分）

#### 2.4.3 检查点恢复

- 可选功能（`--checkpoint` 标志启用）
- 基于 LangGraph 的 SQLite 检查点
- 每股票独立 SQLite 数据库
- 崩溃后可从中断的步骤恢复
- 成功完成后自动清除

### 2.5 CLI 交互界面

#### 2.5.1 交互式配置

CLI 必须引导用户完成以下配置步骤：
1. 输入股票代码（支持交易所后缀如 CNC.TO、7203.T）
2. 选择分析日期（YYYY-MM-DD 格式）
3. 选择输出语言（11 种预设 + 自定义）
4. 选择分析师类型（多选）
5. 选择研究深度（影响辩论轮数：浅层 1 轮 / 中等 3 轮 / 深度 5 轮）
6. 选择 LLM 提供商
7. 选择快速思考模型
8. 选择深度思考模型
9. 配置提供商特定参数（reasoning_effort / effort / thinking_level）

#### 2.5.2 实时运行显示

- 使用 Rich 库实现终端 UI
- 实时显示智能体状态（等待中 / 运行中 / 已完成）
- 滚动消息日志（含工具调用记录）
- 当前分析报告（显示最新完成的报告段落）
- 统计面板：完成智能体数、LLM 调用数、工具调用数、Token 消耗、报告数、耗时

#### 2.5.3 报告保存

- 支持将完整分析保存到磁盘
- 按股票代码分目录组织
- 生成具体的交易日期命名的子目录（如 `2026-01-15/`）
- 按阶段分文件保存：分析师报告、研究辩论、交易决策、风险辩论、投资组合决策
- 生成 `complete_report.md` 完整报告

### 2.6 Python SDK 使用

```python
from tradingagents.graph.trading_graph import TradingAgentsGraph
from tradingagents.default_config import DEFAULT_CONFIG

config = DEFAULT_CONFIG.copy()
config["llm_provider"] = "deepseek"
config["deep_think_llm"] = "deepseek-reasoner"
config["quick_think_llm"] = "deepseek-chat"

ta = TradingAgentsGraph(debug=True, config=config)
_, decision = ta.propagate("NVDA", "2026-01-15")
print(decision)  # Buy / Overweight / Hold / Underweight / Sell
```

### 2.7 REST API 服务

- **FastAPI REST API**：`service/` 目录下的独立 FastAPI 应用，将 TradingAgents 能力暴露为 HTTP API
- 端点：`POST /api/v1/analysis` 提交分析任务，`GET /api/v1/analysis/{job_id}` 查询状态
- 支持 SSE 流式推送分析进度（`GET /api/v1/analysis/{job_id}/stream`）
- SQLite 数据库记录分析任务（通过 SQLAlchemy ORM）
- 报告输出：Markdown + HTML（单阶段文件 + 完整合并文件 + 索引页面），可通过 `GET /api/v1/analysis/{job_id}/view` 查看

### 2.8 Docker 部署

- 提供标准 Dockerfile（多阶段构建，非 root 用户运行）
- 提供 docker-compose.yml（含 Ollama 本地模型 profile）
- 支持 `.env` 文件注入 API 密钥
- 持久化 `~/.tradingagents` 卷

## 3. 非功能需求

### 3.1 安全性

- **路径遍历防护**：股票代码在写入文件系统前必须通过 `safe_ticker_component()` 验证
  - 正则限制字符集：`[A-Za-z0-9._\-\^]`
  - 拒绝长度超过 32 字符的输入
  - 拒绝纯点号输入（如 `..`、`...`）
- 双来源风险：股票代码既来自用户 CLI 输入，也来自 LLM 工具调用的返回（提示注入风险）
- API 密钥通过环境变量注入，不硬编码

### 3.2 可靠性

- 检查点恢复：崩溃后可从中断步骤继续，不丢失已完成的分析
- 原子写入：决策日志使用临时文件 + `os.replace()` 保证写入过程崩溃不损坏数据
- 幂等性：同股票同日期的决策不会重复写入决策日志
- 数据源回退：主数据源不可用时自动切换到备用数据源
- 结构化输出回退：提供商不支持 structured output 时回退到自由文本生成

### 3.3 可扩展性

- LLM 提供商工厂模式：新增提供商只需添加一个 `BaseLLMClient` 子类
- 数据源可插拔：新增数据源只需实现对应方法并在 `VENDOR_METHODS` 注册
- 智能体模块化：每个智能体是独立的工厂函数（`create_xxx(llm)`），返回 LangGraph 节点
- 分析师可选择性启用/禁用

### 3.4 可观测性

- Python logging 模块标准日志
- CLI 实时状态显示（Rich Live Display）
- LLM/Tool 调用统计回调（StatsCallbackHandler）
- 完整分析状态保存为 JSON 文件
- 调试模式：`TradingAgentsGraph(debug=True)` 流式输出每一步

### 3.5 跨平台

- Python 3.10+
- Linux / macOS / Windows（v0.2.4 修复 Windows UTF-8 编码）
- Docker 跨平台部署

## 5. 约束与限制

### 5.1 技术约束

- 依赖 LangGraph/LangChain 生态（LLM 抽象、图编排、检查点）
- 数据时效性依赖第三方数据源（yfinance、Alpha Vantage）的可用性
- LLM 调用成本和延迟是主要瓶颈（每次分析数十到数百次 LLM 调用）

### 5.2 业务约束

- 仅用于研究目的，不作为投资建议
- 交易表现在不同市场条件下差异显著
- 分析质量高度依赖所选 LLM 模型的能力

### 5.3 已知限制

- 回测存在前视偏差风险（代码中已通过日期过滤缓解，但依赖数据源提供准确的时间戳）
- 决策日志仅在同股票下次运行时解析（跨股票待处理条目不会自动解析）
- 检查点恢复仅支持同股票同日期（换日期会创建新 thread）
